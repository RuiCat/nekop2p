# 区域节点共识协议 — 技术设计文档

> 版本: 1.1.0 | 日期: 2026-07-18 | 状态: 🟢 Phase A-D 已实现
>
> 基于虚拟空间坐标 + 双网格叠加 + 银行间结算模型的分片共识协议。
> 替代全局 PBFT，将 O(n²) 通信降至 O(1)。

---

## 一、设计动机

### 1.1 现有共识的瓶颈

```
PBFT 全局共识:
  每笔交易 → 全网广播 → 收集 2/3+ 验证者投票 → 提交
  消息复杂度: O(n²)
  1000 节点: ~1,000,000 条消息/区块
```

### 1.2 区域共识的核心思想

```
用【社会关系→虚拟坐标→区域分片】替代全局共识:
  每笔交易 → 4 方参与 (双方用户 + 双方信任节点)
  消息复杂度: O(1)
  1000 节点: 4 条消息/交易
```

## 一.5 架构分层：执行层 + 记录层

### 职责分离

```
全局链 (记录层) — 只写不验, 不可篡改的审计日志
  ✅ 存储所有已确认交易的摘要
  ✅ 存储区域状态根 Merkle Proof  
  ✅ 存储跨区交易索引
  ✅ 不可篡改的审计轨迹
  ❌ 不参与交易排序
  ❌ 不参与交易验证
  ❌ 不需要全网共识

区域节点 (执行层) — 串行处理 + 交叉审查
  ✅ 交易排序 + 串行处理
  ✅ 余额验证 + 防双花
  ✅ 四方交叉审查
  ✅ 每 N 笔批量提交到全局链
```

### 批量提交流程

```
区域节点每 N 笔交易 (如 100 笔) 向全局链提交一次:

提交内容:
  {
    区域ID: "A-1a",
    起始序号: 5001,
    结束序号: 5100,
    交易摘要: Merkle(5001..5100),
    状态根:   Merkle(最终余额表),
    跨区引用: [ {tx: 5042, partner: "B-3c"} ]
  }

全局链验证 (仅做格式检查, 不验证交易内容):
  ✅ 序号连续性: endSeq - startSeq + 1 == N
  ✅ 状态根非空
  ✅ 跨区引用格式正确
  
全局链 Block:
  Block_K { 
    region_batches: [
      { region: "A-1a", seq: 5001-5100, stateRoot: 0x..., crossRefs: [...] },
      { region: "B-3c", seq: 8001-8100, stateRoot: 0x..., crossRefs: [...] },
    ]
  }
```

### 为什么这样设计

```
纯全局链:    每笔交易都要全局共识 → O(n²) → 极慢
纯区域链:    无审计轨迹 → 无法追责 → 不透明
本模型:      区域处理(快) + 全局记录(审计) = 兼顾性能与可追溯

类比:
  区域节点 = 银行柜台 (处理交易)
  全局链   = 央行清算中心 (记录账本)
  
  柜台处理实时交易, 清算中心定期批量记账
  柜台之间通过交叉审查互信, 清算中心不参与验证
```


## 二、虚拟坐标系统

### 2.1 坐标派生

```
用户坐标 = f(邀请人坐标, 信任距离, 时间, 随机偏移)

创世用户 (原点):
  Coord(0, 0)  ← 首个种子邀请者

普通用户:
  X = inviter.X + cos(angle) × trustDist × baseUnit
  Y = inviter.Y + sin(angle) × trustDist × baseUnit
  
  angle = Hash(inviter.ChainID || invitee.ChainID) → 归一化到 [0, 2π)
  trustDist = 1 (直接好友) 或 2 (好友的好友)
  baseUnit = 可调参数 (默认 1.0)
```

### 2.2 数据结构

```go
type SpatialCoord struct {
    X, Y       float64  // 虚拟坐标
    OriginID   ChainID  // 邀请人 (坐标锚点)
    TrustDist  uint8    // 到原点的信任距离
    DerivedAt  int64    // 坐标生成时间
}

func DeriveCoord(inviter SpatialCoord, invitee ChainID, trustDist uint8) SpatialCoord {
    seed := sha256.Sum256(append(inviter.OriginID[:], invitee[:]...))
    angle := float64(binary.BigEndian.Uint64(seed[:8])) / float64(^uint64(0)) * 2 * math.Pi
    
    return SpatialCoord{
        X: inviter.X + math.Cos(angle) * float64(trustDist),
        Y: inviter.Y + math.Sin(angle) * float64(trustDist),
        OriginID: inviter.OriginID,
        TrustDist: inviter.TrustDist + trustDist,
        DerivedAt: time.Now().Unix(),
    }
}
```

## 三、双网格叠加模型

### 3.1 两个独立网格

```
网格 1 (Grid₁):
  gridSize₁ = G₁
  offset₁   = (0, 0)
  angle₁    = 0°

网格 2 (Grid₂):
  gridSize₂ = G₂  (≠ G₁, 如 G₁×1.3)
  offset₂   = (G₁/3, G₁/3)
  angle₂    = 30°

区域 ID:
  RegionID(grid, x, y) = Hash(grid || floor(x/gridSize) || floor(y/gridSize))[:8]
```

### 3.2 用户-区域映射

```
用户 U 的坐标 (x, y):

网格 1 区域: RegionID(1, x', y')  其中 x'=(x+offset₁.x)/gridSize₁
网格 2 区域: RegionID(2, x'', y'') 其中 x''=(x×cosθ - y×sinθ + offset₂.x)/gridSize₂

关键性质:
  网格1 和 网格2 的用户集合几乎完全不相交
  → 控制了网格1某区域的攻击者无法同时控制网格2对应区域
```

### 3.3 不重叠性证明

```
设网格1有 N₁ 个区域, 网格2有 N₂ 个区域

网格1 的某区域 R₁ᵢ 包含 k₁ 个用户
网格2 的某区域 R₂ⱼ 包含 k₂ 个用户

R₁ᵢ ∩ R₂ⱼ = ∅  的概率:
  P(不相交) ≈ 1 - (k₁·k₂)/(N₁·N₂)

当 G₁ ≠ G₂ 且 offset₁ ≠ offset₂ 时:
  两个划分方案独立, 用户同时落入 R₁ᵢ 和 R₂ⱼ 的概率 = (k₁/N₁)·(k₂/N₂) ≈ 0
```

## 四、区域节点

### 4.1 节点职责

```
区域节点 = 该区域的微型共识引擎 + 流量承载枢纽

职责:
  1. 流量承载   — 区域内所有交易经过本节点
  2. 串行排序   — 每笔交易分配单调递增序号
  3. 余额维护   — 维护区域内用户的余额状态
  4. 跨区协调   — 与目标区域节点原子协调跨区交易
  5. 双活冗余   — 同一区域两个节点互备
  6. 交叉审查   — 交易双方节点互相验证
```

### 4.2 节点选举

```
区域节点选举条件:
  1. TrustWeight ≥ 阈值 (如 500)
  2. 坐标属于本区域
  3. 在线率 ≥ 95%
  4. 历史上无罚没记录

选举算法:
  1. 收集本区域内所有合格节点
  2. 按 TrustWeight 降序排列
  3. 取前 2 名作为区域节点
  4. 第 3-5 名作为影子候补

故障转移:
  区域节点心跳间隔 = 2 秒
  连续 3 次无响应 → 影子候补自动顶替
  顶替后同步最新状态快照
```

### 4.3 数据结构

```go
type RegionNode struct {
    RegionID   string
    GridIndex  int                // 0=网格1, 1=网格2
    Members    map[ChainID]*MemberState
    Chain      []*RegionBlock     // 区域链
    TxSeq      uint64             // 交易序号
    StateRoot  [32]byte           // Merkle 状态根
}

type MemberState struct {
    ChainID    ChainID
    Balance    uint64
    LockedAmt  uint64             // 交易中锁定金额
    Coord      SpatialCoord       // 用户坐标
    OtherNode  RegionID           // 用户在另一个网格的区域节点
}

type RegionBlock struct {
    Height     uint64
    PrevHash   [32]byte
    Tx         *Transaction
    Timestamp  int64
    RegionSig  []byte             // 区域节点 Ed25519 签名
}
```

## 五、交易协议

### 5.1 同区域交易 (A 和 B 在同一区域)

```
Phase 1: 发起
  A → R (区域节点): { to: B, amount }
  R: 生成 TxID = 区域ID-seq→seq++
  R → A: { txID }

Phase 2: 余额检查
  R: if A.balance < amount → 拒绝
  R: A.balance -= amount
  R: B.balance += amount

Phase 3: 确认
  R → A: { txID, done }
  R → B: { txID, from: A, amount, done }
  R: 写入区域链

延迟: ~1ms (本地处理)
```

### 5.2 跨区域交易 (四方协议)

```
前提: A 选择 RA₁ (网格1), B 选择 RB₂ (网格2)

Phase 1: A 申请
  A → RA₁: { to: B, amount, B的区域节点: RB₂ }
  RA₁: 验证 A ∈ RA₁区域
  RA₁: 生成 TxID, 锁定 A.balance -= amount
  RA₁ → A: { txID, locked }

Phase 2: 银行间交叉审查
  RA₁ → RB₂: {
    txID, from: A, to: B, amount,
    RA₁签名, A的坐标, A的区域验证
  }
  RB₂: 验证 RA₁ 签名 ✓
  RB₂: 验证 B ∈ RB₂区域 ✓
  RB₂: 验证 A 在 RA₁ 的合法用户集中 ✓

Phase 3: B 确认
  RB₂ → B: { txID, from: A, amount, 待确认 }
  B → RB₂: { txID, B签名, 同意 }
  RB₂: B.balance += amount

Phase 4: 结算完成
  RB₂ → RA₁: { txID, RB₂签名, 已结算 }
  RA₁: 永久扣除 A.balance (解锁→扣款)
  RA₁ → A: { txID, 完成 }

RA₁ 写入区域链: Block_N { cross: true, partner: RB₂, partnerTx: "Y-M" }
RB₂ 写入区域链: Block_M { cross: true, partner: RA₁, partnerTx: "X-N" }

每 N 笔交易后, 区域节点批量提交到全局链:
  RA₁ → 全局链: { region: "A-1a", seq: N..N+99, stateRoot, crossRefs }
  RB₂ → 全局链: { region: "B-3c", seq: M..M+99, stateRoot, crossRefs }
```

### 5.3 选择策略

```
用户 A 选择信任节点的策略:

1. 优先选择: 在线 + 延迟低的节点
2. 其次选择: 历史表现好的节点 (高在线率)
3. 负载均衡: 如果当前节点队列 > 阈值, 切换到备选节点

B 选择时独立决策，不依赖 A 的选择
→ A 选 RA₁ (网格1), B 选 RB₂ (网格2) 是最常见情况
→ 但也可能: A 选 RA₂ (网格2), B 选 RB₁ (网格1)
→ 无论如何, 两个节点来自不同网格 → 用户集不重叠
```

## 六、链数据同步

### 6.1 同步模型

```
不再有"一条全局链", 而是:
  每条区域链独立维护
  跨区交易在双方区域链中各存一份 (互相引用)

同步 = 追本区域的链 + 验证跨区引用的一致性
```

### 6.2 五种同步场景

```
场景1: 区域节点故障转移
  RA₁ 宕机 → 影子 RA₁' 接管
  RA₁' → RA₂: "同步最新状态"
  RA₂ 返回: [完整区域链 + 最新状态快照]
  RA₁': 验证 Merkle 根 → 接管服务

场景2: 用户换设备恢复
  A → RA₁: "给我我的历史"
  RA₁ 返回: A 的所有交易 + 当前余额
  A → RA₂: 交叉验证
  RA₂: 对比自己的记录 → 一致则确认

场景3: 新用户加入区域
  B → RB₁: "加入, 这是邀请凭证"
  RB₁: 验证凭证 → 计算坐标 → 加入成员列表
  RB₁ → RB₂: "新成员 B, 初始余额 X"
  RB₂: 记录 B 的跨网格映射

场景4: 跨区交易同步
  跨区交易完成后:
  RA₁ 链: { ..., cross: true, partner: RB₂ }
  RB₂ 链: { ..., cross: true, partner: RA₁ }
  两条链互相引用 → 不可抵赖

场景5: 区域分叉恢复
  RA₁ 和 RA₂ 的链出现分歧:
  取共同前缀
  分歧后:
    同区域交易 → 序号大者胜
    跨区交易 → 查询对方区域的记录裁决
```

### 6.3 状态快照

```
每 N 笔交易 (默认 1000) 生成快照:

type Snapshot struct {
    Height       uint64
    Balances     map[ChainID]uint64
    MerkleRoot   [32]byte
    RegionSig    []byte
}

快照用途:
  1. 新节点快速同步 (不追全部历史)
  2. 状态验证 (两个网格的快照 Merkle 根对比)
  3. 跨区交易验证 (检查对方区域的快照)
```

## 七、安全分析

### 7.1 攻击模型

| 攻击 | 防御 | 安全等级 |
|:---|:---|:---|
| 女巫攻击 (伪造坐标) | 坐标由邀请链推导, 需伪造社交链 | ★★★★★ |
| 区域节点作恶 | 需同时控制2个不同网格节点 | ★★★★★ |
| 交易重放 | TxID 区域唯一 + 序号单调 | ★★★★★ |
| 余额双花 | 区域节点串行处理, 无并发 | ★★★★★ |
| 共谋攻击 | 双网格用户集不重叠, 无共谋动机 | ★★★★★ |
| 区域节点宕机 | 双活冗余 + 影子候补 | ★★★★☆ |
| 跨区交易伪造 | 双链互相引用, 单方无法伪造 | ★★★★★ |
| 状态快照伪造 | 双网格 Merkle 根交叉验证 | ★★★★☆ |

### 7.2 拜占庭容错

```
假设: 攻击者控制了某个区域的节点

攻击者能做到:
  ✗ 伪造交易 → 对方节点交叉审查会拒绝
  ✗ 双花余额 → 串行处理天然防止
  ✗ 回滚交易 → 跨区交易有对方链引用, 单方回滚会断裂
  ✗ 审查用户 → 用户可以切换到另一个网格的节点

攻击者做不到:
  ✗ 同时控制两个不同网格的节点
  ✗ 伪造跨区交易 (双方链互相引用)
  ✗ 篡改状态快照 (双方 Merkle 根必须一致)
```

## 十、总结

区域节点共识协议通过四个核心机制构建了高性能、可审计的共识系统:

1. **虚拟坐标分片** — 社会关系→空间映射, 天然防女巫
2. **双网格叠加** — 用户集不重叠, 消除共谋动机
3. **银行间结算** — 4方交易协议, 交叉审查 = 共识
4. **全局链记录** — 批量提交 + Merkle Proof, 审计可追溯

通信复杂度: O(n²) → O(1)
安全假设: 2/3+ 诚实 → 双网格独立 + 串行处理 + 不可篡改审计
