## 二、设计哲学

### 2.1 可玩性优先而非金融化优先

| | 传统 GameFi | 明域游戏 |
|:---|:---|:---|
| **目的** | 玩是为了赚 (Play to Earn) | 为乐付费 (Pay for Fun) / 为快付费 (Pay for Speed) |
| **资产** | 核心是 NFT 炒作 | 核心是游戏内容 |
| **经济** | 暴露于外部代币波动 | 内部经济隔离，独立调控 |
| **服务器** | 中心化，可能关停 | 链上状态机，不可停止 |

### 2.2 规则不可篡改 · 数据永存

游戏逻辑作为链上状态机运行。开发者离开后游戏继续存在。规则修改需经过公告期 + 链上留痕。

### 2.3 主体性与客观性分离

在核心掉落算法中：
- **主体性** = 玩家的个人努力和技巧 (K_i)
- **客观性** = 所有玩家共同创造的环境 (P)

奖励由个人因素在一个客观给定的价值天花板内获取。奖励个人奋斗，防止个人垄断。

## 三、游戏状态机

### 3.1 核心模型

每个游戏 = 一个链上状态机。状态迁移由玩家提交的交易触发，由游戏规则验证。

```
GameState:
  game_id:        唯一标识
  author_id:      游戏作者 chain_id
  version:        当前规则版本
  pending_version: 待生效规则版本 (公告期内)
  
  assets:         asset_id → Asset
  players:        player_id → PlayerState
  regions:        region_id → RegionState  
  activities:     activity_id → ActivityState
  market:         商城/交易所状态
  
  params:         GameParams (作者可调参数)
  treasury:       游戏金库
```

### 3.2 状态迁移

```
玩家提交 action → 链上验证规则 → 迁移状态 → 生成 event

action:
  - KillMonster    { player_id, monster_id, region_id, random_seed_commitment }
  - CraftMaterial   { player_id, recipe_id, ingredients[] }
  - Trade           { seller_id, buyer_id, asset_id, price }
  - PurchaseMall    { player_id, item_id }
  - EnterRegion     { player_id, region_id }
  - LeaveRegion     { player_id, region_id }
```

```go
type GameStateMachine struct {
    game   *GameDefinition
    state  *GameState
    rand   *DistributedRandomBeacon  // 分布式随机信标
}

func (gsm *GameStateMachine) Apply(action Action) (*StateTransition, error) {
    // ① 验证 action 合法性 (规则引擎)
    if err := gsm.game.Rules.Validate(action, gsm.state); err != nil {
        return nil, err
    }
    
    // ② 需要随机数 → 查询分布式随机信标的当前轮次结果
    if action.NeedsRandom() {
        action.RandomResult = gsm.rand.GetRound(action.Round)
    }
    
    // ③ 执行状态迁移
    transition := gsm.game.Rules.Execute(action, gsm.state)
    
    // ④ 链上记录
    transition.RecordOnChain()
    
    return transition, nil
}
```

## 四、资产系统

### 4.1 资产类型

| 类型 | 说明 | 示例 |
|:---|:---|:---|
| **素材 (Material)** | 可铸造/掉落/消耗的基本单元 | 铁矿石、龙鳞、魔法粉尘 |
| **装备 (Equipment)** | 由素材铸造的成品 | 屠龙刀、秘银甲 |
| **消耗品 (Consumable)** | 一次性使用 | 血瓶、传送卷轴 |
| **外观 (Cosmetic)** | 纯装饰性、不影响属性 | 皮肤、特效 |
| **凭证 (Certificate)** | 权益证明 | 入场券、VIP资格 |
| **金币 (Gold)** | 游戏内货币 | 通过充值/交易获得 |

### 4.2 资产铸造

```
铸造 = 消耗素材 + 满足配方 → 生成装备

CraftMaterial:
  输入: player_id, recipe_id, ingredients[(asset_id, quantity)]
  
  验证:
    ① 玩家持有足量素材
    ② 满足配方要求的素材类型和数量
    ③ 玩家技能等级满足配方要求
    
  执行:
    ① 销毁素材
    ② 生成装备 (属性由配方 + 铸造随机性决定)
    ③ 装备所有权归属玩家
    ④ 链上记录铸造事件
```

### 4.3 资产所有权

```
Asset:
  asset_id:   唯一标识 (内容哈希)
  type:       素材/装备/消耗品/外观/凭证
  metadata:   { name, description, attributes, image_hash }
  owner:      当前持有者 chain_id
  history:    [铸造→持有者A→交易→持有者B→...]
  proof:      存在性证明 (链上锚定)

所有权转移 = 链上交易，不可回滚
```

## 五、掉落分配算法（核心创新）

### 5.1 双维度模型

```
单次掉落事件的价值分配:

  总价值 V_total = 区域总产出 (由P决定)
  
  玩家i获得的掉落价值:
    V_i = (K_i / ΣK) × V_total
    
  其中:
    P  = 参与击杀的玩家数 (客观维度，反映区域活跃度)
    K_i = 玩家i的击杀贡献 (主体性维度，反映个人努力)
    
    V_total = f(P)  — 随参与人数增加而增加，但有上限
```

### 5.2 详细公式

```go
func CalculateLootDistribution(region *RegionState, participants []PlayerContribution) []LootAllocation {
    P := len(participants)
    
    // 客观维度: 区域产出价值天花板
    // f(P) = base × (1 + α × log(P+1))  ← 对数增长，边际递减
    baseValue := region.BaseLootValue
    alpha := region.LootScalingFactor  // 典型值: 0.3 ~ 0.5
    V_total := baseValue * (1.0 + alpha * math.Log(float64(P)+1))
    V_total = min(V_total, region.MaxLootValue)  // 天花板
    
    // 主体性维度: 各玩家击杀贡献
    totalK := 0
    for _, p := range participants {
        totalK += p.KillContribution
    }
    
    // 分配
    allocations := make([]LootAllocation, P)
    for i, p := range participants {
        share := float64(p.KillContribution) / float64(totalK)
        value := uint64(share * V_total)
        
        // 从这个价值池中按概率掉落具体物品
        items := region.LootTable.Roll(value)
        
        allocations[i] = LootAllocation{
            PlayerID: p.PlayerID,
            Value:    value,
            Items:    items,
        }
    }
    
    return allocations
}
```

### 5.3 掉落表 (Loot Table)

```
LootTable:
  每个区域有一个加权掉落表
  
  [稀有度]  →  [权重]  →  [具体物品池]
  
  传说       →  0.5%    →  {龙鳞×1, 远古符文×1, ...}
  史诗       →  2.0%    →  {秘银矿×3, 魔法水晶×1, ...}
  稀有       →  8.0%    →  {精铁矿×5, 附魔卷轴×1, ...}
  精良       →  20.0%   →  {铁矿石×10, 皮革×3, ...}
  普通       →  69.5%   →  {铜矿石×15, 亚麻布×5, ...}
  
  每次掉落 = 从满足价值预算的稀有度范围内随机选取物品
```

### 5.4 安全性

防止单方面操纵掉落:

```
掉落三要素:
  ① 区域状态 (P): 链上客观数据，所有人可见，不可伪造
  ② 击杀贡献 (K_i): 每个玩家提交 + 其他参与者交叉验证
  ③ 随机种子: 来自分布式随机信标，任何单一玩家无法预测或操控

攻击分析:
  伪造击杀贡献 → 需多数参与者合谋 → 与利益冲突 (合谋者分得的奖励减少)
  预测随机种子 → 分布式随机信标: 只要1个参与者诚实即不可预测
  单刷区域 → P=1 → V_total = base×(1+α×log(2)) → 收益有限，多人协作更优
```

## 六、游戏内经济

### 6.1 三层经济

```
┌─────────────────────────────────────┐
│  第三层: 外部代币 (链上资产)          │
│  · 通过单向兑换桥与内部金币隔离       │
│  · 充值: 外部代币 → 游戏金币          │
│  · 提现: 游戏金币 → 外部代币 (限流)   │
├─────────────────────────────────────┤
│  第二层: 游戏金币                     │
│  · 充值获得 / 交易赚取 / 活动奖励     │
│  · 商城消费 / 交易所购买 / 铸造消耗   │
│  · 作者可调控增发/回收                │
├─────────────────────────────────────┤
│  第一层: 游戏素材与装备                │
│  · 掉落获取 / 铸造生成 / 交易流转     │
│  · 绑定玩家公钥，链上可查             │
│  · 价值由游戏内供需决定               │
└─────────────────────────────────────┘
```

### 6.2 单向兑换桥

```
充值 (外部 → 内部):
  用户销毁外部代币 → 链上证明 → 游戏金币增加
  无限额，无审查

提现 (内部 → 外部):
  用户消耗游戏金币 → 链上证明 → 外部代币释放
  有限额 (防挤兑) + 有冷却期 (防套利)
```

### 6.3 作者经济工具

作者可调控的参数 (全部链上公开):

```go
type EconomicParams struct {
    // 货币调控
    GoldPerRecharge    float64  // 1外部代币 = ?游戏金币 (充值汇率)
    GoldPerWithdraw    float64  // ?游戏金币 = 1外部代币 (提现汇率)
    WithdrawCooldown   int64    // 提现冷却 (秒)
    MaxDailyWithdraw   uint64   // 每人每日提现上限
    
    // 掉落调控
    GlobalDropRate     float64  // 全局掉落倍率 (0.5~2.0)
    LootScalingAlpha   float64  // 区域产出增长系数
    
    // 铸造调控
    CraftCostMultiplier float64 // 铸造消耗倍率
    
    // 商城调控
    MallPrices         map[ItemID]uint64  // 商城价格表
    
    // 活动
    ActiveEvents       []EventConfig      // 进行中的活动
}
```

**权力约束:**
- 任何参数修改需公告期 (≥ 72小时)
- 公告期内参数不可生效
- 每次修改链上留痕 (改了什么、何时改、谁改的)
- 紧急修改需附带紧急理由声明 (同样留痕)

## 七、作者治理模型

### 7.1 作者角色

```
游戏作者 = 央行行长 + 活动策划 + 规则制定者

拥有:
  ✅ 调整经济参数 (公告期内)
  ✅ 新增素材/装备/配方
  ✅ 举办活动 (限时掉落、任务、排行榜)
  ✅ 发布新版本规则 (公告期内)
  ✅ 接收商城收入
  ✅ 接收铸造手续费

不能:
  ❌ 单方面删除玩家资产
  ❌ 篡改历史掉落记录
  ❌ 不经公告修改规则
  ❌ 关闭游戏 (状态机永久运行)
  ❌ 修改分布式随机信标结果
```

### 7.2 治理透明机制

```
每次操作的三层记录:

① 公告期 (事前):
   操作类型 + 参数变化 + 生效时间
   链上广播，玩家可见

② 执行 (事中):
   交易签名 + 时间戳 + 操作内容
   链上永久记录

③ 审计 (事后):
   任何玩家可独立重放验证
   操作是否符合公告内容
   操作是否在作者权限范围内
```

## 八、分布式随机信标在游戏中的应用

### 8.1 应用场景

| 场景 | 随机需求 | 使用方式 |
|:---|:---|:---|
| **怪物掉落** | 每击杀一次 | 信标轮次号绑定到击杀事件 |
| **素材铸造** | 属性随机 (如攻击力±5%) | 铸造时取当前轮次 |
| **宝箱开启** | 每开一次 | 绑定到开启事件 |
| **活动抽奖** | 每天一次 | 取当天最后一轮 |
| **Boss刷新** | 不定时 | 上一次击杀后第N轮触发 |

### 8.2 防作弊

```
问题: 玩家可能在提交commitment后，看到随机结果不利就拒绝reveal

解决方案:
  ① 提交阶段: 玩家提交 commitment + 押金
  ② 揭示阶段: 限时揭示，超时押金被没收
  ③ 使用阶段: 随机结果应用到掉落
  
  玩家拒绝揭示 → 押金没收 + 该玩家贡献记为0
  其余诚实玩家正常分配掉落
```

## 十、性能模型

### 10.1 链上 vs 链下分工

```
链上 (低频, 高价值):
  · 资产所有权转移
  · 掉落分配结果锚定
  · 铸造事件
  · 经济参数变更
  · 公会创建/加入/退出

链下 (高频, 低价值):
  · 玩家移动/动作
  · 实时战斗计算
  · 聊天消息
  · UI状态

锚定方式:
  链下操作定期批量提交 merkle root 到链上
  玩家可随时用 merkle proof 验证链下数据未被篡改
```

### 10.2 适配游戏类型

| 适配度高 | 适配度中 | 适配度低 |
|:---|:---|:---|
| 策略游戏 | RPG冒险 | 动作游戏 (延迟敏感) |
| 卡牌游戏 | 模拟经营 | FPS射击 |
| 回合制对战 | 放置养成 | 格斗游戏 |
| 异步对战 | MMORPG (非实时) | 竞速游戏 |
| 大逃杀 (回合制) | 塔防 | MOBA |
