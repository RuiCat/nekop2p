## 🟢 Phase D: 金融工程优化 (2026-07-06 新增)

> 基于 [金融工程分析](./金融工程分析.md) 提出的两项优化建议。

### D1. 延期准备金拨备 (Deferred Provisioning)
- **文件**: `x/brightchain/keeper/recourse.go` (新建)
- **现状**: 递归追偿瞬间全额扣除担保人 Bond，极端事件中可能引发多米诺骨牌效应
- **目标**: 引入基于历史纪元 (Epoch) 的分级递进扣除机制
  - 观察期 3 Epoch，递进比例 [30%→70%→100%]
  - 申诉窗口 10,000 blocks
  - 连续 50 Epoch 无违约的节点豁免加速追偿
- **金融依据**: 类比巴塞尔 III 逆周期资本缓冲 (Counter-Cyclical Capital Buffer)
- **工作量**: 8h
- **状态**: ⏳ 未开始
- **风险缓解**: 将惩罚从瞬时脉冲展开为时间序列，为关联节点提供资产隔离和申诉窗口

### D2. Shadow Claims 二级市场 (Inkwell 衍生品过桥)
- **文件**: `x/brightchain/shadow_claim.go` (新建), `zkcircuits/shadow_claim/` (新建)
- **现状**: Inkwell 锁仓 1~90 天期间资金完全冻结，造成隐形流动性折价
- **目标**: 允许发行带 ZK 证明的远期影子凭证 (Shadow Claims) 并在明域二级市场转让
  - ZK 绑定: 所有权 + 释放日期 + 面值
  - nullifier 防双花
  - 单笔锁仓最多 1 张凭证 (防过度杠杆化)
- **金融依据**: 类比资产支持商业票据 (ABCP) 和远期利率协议 (FRA)
- **工作量**: 16h
- **状态**: ⏳ 未开始
- **收益**: 释放锁仓资金效率，买卖双方各取所需 (即时流动性 vs 时间套利)

### D3. 掉落算法动态 P 值治理
- **文件**: `x/brightchain/keeper/game_economy.go` (新建)
- **现状**: 掉落算法全局上限 P 为静态参数，用户量激增时可能引发通货紧缩
- **目标**: 实现 P 值的链上治理机制
  - 基于活跃用户数 N 的自适应调节: $P = P_{base} \times \sqrt{N / N_{base}}$
  - 治理提案 + 投票周期 ≥ 7 天
  - 变更后有 30 天渐变过渡期
- **工作量**: 6h
- **状态**: ⏳ 未开始
- **风险缓解**: 防止「生产力陷阱」——越多人参与，单个人产出越低

### E4~E6+E10: 中观层 · 社区异步清算 (P2)

#### E4. 虚拟根网图拓扑计算 (VRG-04)
- **文件**: `x/vrg/topology.go` (新建)
- **现状**: 无跨社区拓扑距离计算方法
- **目标**: 基于创世邀请链的 LCA (最低共同祖先) 计算任意两社区的距离
  - $d(V_A, V_B) = \text{depth}(V_A) + \text{depth}(V_B) - 2 \times \text{depth}(\text{LCA}(V_A, V_B))$
  - 安全系数: $S = f(d, \text{Bond质押额})$
  - 若 $d \le$ 阈值 → 允许宏观清算；否则拒绝或要求额外质押
- **依赖**: IAVL 存储 (A2) + 创世链数据
- **工作量**: 8h
- **状态**: ⏳ 未开始

#### E5. Namespace 隔离 (VRG-05)
- **文件**: `app/app.go` (修改), `x/vrg/namespace.go` (新建)
- **现状**: ABCI 层无应用命名空间隔离，所有交易混在同一状态树
- **目标**: 实现 NamespaceID 路由 + 状态树隔离
  - 每个应用分配唯一 `NamespaceID` (uint16)
  - ABCI 胶水层按 Namespace 路由交易至独立子状态树
  - 跨 Namespace 交易需通过外交通道 (TrustEdge)
- **依赖**: Phase A 共识 + 存储
- **工作量**: 10h
- **状态**: ⏳ 未开始

#### E6. 外交通道 (VRG-06)
- **文件**: `x/vrg/diplomacy.go` (新建)
- **现状**: 不同 Namespace 的社区之间无信任桥梁
- **目标**: 跨 Namespace TrustEdge 建立与管理
  - 链上治理提案建立外交通道 (双方质押 Bond)
  - TrustEdge 状态机: `PROPOSED → ACTIVE → FROZEN → SEVERED`
  - 违约时自动 Slashing 双方 Bond 并切断通道
- **依赖**: E5 Namespace 隔离 + 共识
- **工作量**: 12h
- **状态**: ⏳ 未开始

#### E10. 双花冲突检测 (VRG-10)
- **文件**: `zkcircuits/marker/circuit.go` (修改)
- **现状**: marker 电路已实现基础的纪元绑定，但缺乏跨纪元碰撞检测逻辑
- **目标**: 实现跨纪元双花碰撞的自动识别和触发
  - 同一资产在同一 Epoch 内向多个社区发送影子收据 → 瞬间电路冲突
  - 冲突后自动触发递归 Slashing
  - 冲突事件广播至全网 (警示其他社区)
- **依赖**: B1 ZK 集成 + E2 影子收据
- **工作量**: 12h
- **状态**: ✅ 已完成 — `store/proto_wire.go` gob/JSON混合序列化
