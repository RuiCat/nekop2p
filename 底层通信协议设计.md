# 底层通信协议设计

> 版本: 0.1.0 | 状态: 设计阶段 | 实现语言: Go

---

## 一、架构定位

本文档描述**共享网络层**的底层通信协议，为上层双链架构（明链/暗链）提供 P2P 通信基础设施。

```
┌──────────────────────────────────────┐
│   应用层：明域(游戏) / 暗域(金融)     │  ← 后续
├──────────────────────────────────────┤
│   协议层：双链结算 / 混沌池 / ZK      │  ← 后续
├──────────────────────────────────────┤
│   网络层：P2P通信 / 节点发现 / 节点体系 │  ← 【本文档】
├──────────────────────────────────────┤
│   物理层：线下邀请 / 双密钥 / 邓巴数   │  ← 部分涉及
└──────────────────────────────────────┘
```

### 核心设计原则

1. **节点即非信任实体。** 协议层不信任任何转发节点。信任只存在于加密密钥持有者之间。
2. **局部视图，全局不可知。** 每个节点只知道自己的直接连接和好友的好友（半径 ≤ 3）。全局拓扑不存在于任何单一位置。
3. **网络即社交图。** 连接关系由线下邀请担保链决定。边 = 信任关系 = 通信路径。
4. **按需匿名。** 日常直连高效通信；敏感时启用洋葱路由等隐私层（Phase 2+）。

## 二、物理层约束

### 2.1 网络协议

- **仅支持 IPv6。** 每个节点拥有全球可路由 IPv6 地址，无需 NAT 穿透。
- 传输层: TCP（Phase 1），后续可升级 QUIC 多路复用。

### 2.2 链上双密钥对

每个用户在链上持有两个完全无关的密钥对：

| 密钥对 | 链上公开 | 本地私持 | 用途 |
|:---|:---|:---|:---|
| **recv** | `recv_pk` | `recv_sk` | 任何人→我: 用 `recv_pk` 加密，只有我能解 |
| **send** | `send_pk` | `send_sk` | 我→任何人: 我用 `send_sk` 签名，对方用 `send_pk` 验证 |

**密钥文件 (用户自行保管，不在程序目录):**

```
bright_keys.json          dark_keys.json
  recv_sk (Curve25519)      master_secret (32B)
  send_sk (Ed25519)         → anon_id/identity_marker
  → chain_id

两个文件密码学完全独立。持有bright_keys不能追查暗域活动。
```

链上存储 (UserBlock):
```
chain_id → {
  recv_pk, send_pk,        ← 公开密钥
  friends[],                ← 好友列表 (链上存储)
  credit_score, trust_weight, ← 信誉
  ...
}
```

## 三、传输握手协议

### 3.1 协议选择

不使用 TLS。使用 **Noise Protocol Framework**——更轻、握手更少、不依赖证书。

| 场景 | 发起方知接收方公钥? | 接收方知发起方公钥? | Noise 模式 | RTT |
|:---|:---|:---|:---|:---:|
| 好友直连 | ✅ | ✅ | Noise_IK | 0.5 |
| 连接公开节点 | ✅ | ❌ (匿名) | Noise_NK | 0 |
| 反向连接回应 | ✅ | ❌ | Noise_NK | 0 |
| 填充连接 | 不一定 | ❌ | Noise_NK | 0 |

### 3.2 密码套件

```
Noise_IK_25519_ChaChaPoly_BLAKE3:
  DH:       Curve25519
  对称加密: ChaCha20-Poly1305 (软件速度比AES-GCM快3-5倍)
  哈希:     BLAKE3

Noise_NK_25519_ChaChaPoly_BLAKE3:
  同上
```

选用 ChaCha20-Poly1305 的理由：移动设备和低功耗节点无 AES-NI 硬件时，ChaCha20 软件性能显著优于 AES。握手阶段每条消息独立加密，软件速度差异明显。

### 3.3 Noise_IK（好友直连，0.5-RTT）

```
Bob (发起方)                            Alice (响应方)
已知: alice_static_pk                   已知: bob_static_pk

① 生成 bob_ephemeral
   发送: e (bob_ephemeral_pk)
                                       ② 生成 alice_ephemeral
                                         DH1 = DH(als_e, bob_e)
                                         DH2 = DH(als_e, bob_s)
                                         DH3 = DH(als_s, bob_e)
                                         shared = KDF(DH1||DH2||DH3)
                                         发送: e, ee, se, es (加密)
                                       
③ DH1 = DH(bob_e, als_e)
   DH2 = DH(bob_s, als_e)
   DH3 = DH(bob_e, als_s)
   shared = KDF(DH1||DH2||DH3)
   
   ←═══ session_key 建立，双方身份已认证 ═══→
```

IK 模式握手即认证——双方静态密钥参与 DH，不需要额外发身份证明。

### 3.4 Noise_NK（公开节点/反连/填充，0-RTT）

```
Bob (发起方)                            节点X (响应方)
已知: x_static_pk                       未知 Bob 身份

① 生成 bob_ephemeral
   DH1 = DH(bob_e, x_static_pk)
   发送: e, es (用DH1派生的密钥加密)
   [Bob可以在es中直接捎带数据]
                                       ② 生成 x_ephemeral
                                         DH2 = DH(x_e, bob_e)
                                         shared = KDF(DH1||DH2)
                                         发送: e, ee (加密)
                                       
③ DH2 = DH(bob_e, x_e)
   shared = KDF(DH1||DH2)
   
   ←═══ session_key 建立 ═══→
   节点X不知Bob身份（仅知Bob临时公钥）
   Bob确认对方是节点X（X的静态密钥参与DH）
```

### 3.5 Prologue 绑定

握手时混入上下文，防止跨协议攻击：

```go
prologue := SHA256("nekop2p-v1" || chain_genesis_hash || role)
// role = "friend" | "public" | "reverse" | "padding"
```

不同场景不同 prologue → 同密钥对不同 session_key → 防止跨场景重放。

### 3.6 握手后的帧加密

Noise 握手输出两个 CipherState（发送/接收方向），直接用于传输帧加密。帧格式微调：

```
线上字节 (节点间 TCP 连接):
┌────────┬──────────┬──────────────────────┬──────────┐
│  Len   │  Nonce   │      Ciphertext      │ Auth Tag │
│ (2B)   │  (8B)    │      (variable)      │  (16B)   │
│ 大端序  │ 递增计数器 │ ChaCha20-Poly1305加密 │          │
└────────┴──────────┴──────────────────────┴──────────┘
                        ↓ 解密后
┌────────┬────────┬─────────┬───────────┬──────────────┐
│  Ver   │  Type  │  Flags  │ Stream ID │   Payload    │
│ (1B)   │  (1B)  │  (1B)   │  (2B)     │  (variable)  │
└────────┴────────┴─────────┴───────────┴──────────────┘
```

Nonce: 64位（ChaCha20-Poly1305 标准）。外帧开销 2+8+16=26B。

## 四、传输帧格式

### 4.1 分层结构

```
线上字节 (节点间 TCP 连接):
┌────────┬──────────┬──────────────────────┬──────────┐
│  Len   │  Nonce   │      Ciphertext      │ Auth Tag │
│ (2B)   │  (8B)    │      (variable)      │  (16B)   │
│ 大端序  │ 递增计数器 │ ChaCha20-Poly1305加密 │          │
└────────┴──────────┴──────────────────────┴──────────┘
                        ↓ 解密后
┌────────┬────────┬─────────┬───────────┬──────────────┐
│  Ver   │  Type  │  Flags  │ Stream ID │   Payload    │
│ (1B)   │  (1B)  │  (1B)   │  (2B)     │  (variable)  │
└────────┴────────┴─────────┴───────────┴──────────────┘
```

### 4.2 帧类型

| Type | 名称 | 用途 | Phase |
|:---|:---|:---|:---|
| 0x01 | BEACON | 加密信标转发 | 1 |
| 0x02 | DATA | 端到端加密消息 (双棘轮后) | 1 |
| 0x03 | PING | 心跳保活 | 1 |
| 0x04 | PONG | 心跳应答 | 1 |
| 0x05 | CLOSE | 优雅关闭 | 1 |
| 0x06 | ROUTE | 洋葱路由中继 | 2 |

## 五、加密信标发现协议 (Encrypted Beacon Discovery)

### 4.1 用途

节点上线时，通过公开节点向网络注入加密信标，让好友发现自己。信标沿信任图洪泛传播，只有目标好友能解密并反向连接。

### 4.2 包结构

```
┌─────────────────────────────────────────────────────┐
│                  明文外层 (洪泛控制)                  │
│  random_tag(16B) | hop_cnt(1B) | hop_max(1B=7)     │
├─────────────────────────────────────────────────────┤
│                  载荷体 (AES-256-GCM)                │
│  加密内容:                                           │
│    bob_chain_id       (32B)  发送方链上ID            │
│    bob_ipv6           (16B)  回连地址                │
│    bob_port           (2B)   回连端口                │
│    beacon_nonce       (32B)  一次性挑战码            │
│    timestamp          (8B)   签发时间                │
│  + Sign(Bob.send_sk)  (64B)  发送方签名              │
├─────────────────────────────────────────────────────┤
│              密钥包裹槽位 (M 个, 每槽 64B)            │
│  slot[i] = ECDH_KEM(target_i.recv_pk, payload_key)  │
│  M = ceil(实际目标数, 8) × 8                         │
│  未使用槽位: 随机填充                                 │
└─────────────────────────────────────────────────────┘
```

### 4.3 多目标机制

- 一次信标可包含多个目标好友 (通过多个槽位)。
- 载荷体只存一份，用随机对称密钥 `payload_key` 加密。
- 每个槽位用对应目标的 `recv_pk` 包裹 `payload_key`。
- 槽位总数填充到 8 的倍数，隐藏真实好友数量。

### 4.4 传播规则

```
节点收到 BEACON 帧:
  ① 检查 random_tag 是否已见 → 是则丢弃
  ② 尝试每个槽位: ECDH_KEM_Decrypt(my_recv_sk, slot)
     → 成功: 解密载荷体 → 验证签名 → 验证时间戳 → 我是目标
     → 失败: 继续下一槽位
  ③ 不是目标 → hop_cnt++ → 若 ≤ hop_max 则转发给所有已连接节点
```

### 4.5 目标处理流程

```
解密成功 + 我是目标:
  ① 链上查 sender_chain_id → 获取 send_pk
  ② 验证 Sign(send_pk) → 失败则丢弃 (防伪造)
  ③ 验证 timestamp (5分钟窗口) → 过期丢弃 (防重放)
  ④ 验证 beacon_nonce 未消费 → 已消费丢弃 (防窗口内重放)
  ⑤ 检查 sender_chain_id 是否在好友列表 → 否 → 静默丢弃
  ⑥ ✅ 全部通过 → 构造回应包
```

### 4.6 防重放与防伪造

| 攻击 | 防御 |
|:---|:---|
| 伪造信标 (以他人身份) | 需要 send_sk 签名 → 没有 → 签名验证失败 |
| 篡改信标内容 | GCM 认证标签 → 篡改即失效 |
| 长期重放 (过期信标) | timestamp 超出5分钟窗口 → 丢弃 |
| 窗口内重放 | beacon_nonce 已消费 → 丢弃 |
| 垃圾信标 | 好友列表过滤 + 可拉黑 chain_id |

## 六、反向连接认证协议

### 5.1 用途

好友解密信标后，向发送方发回加密回应包，建立 TCP 直连。

### 5.2 回应包结构

```
UDP 直连 (Alice → Bob):

┌──────────────────────────────────────────────────┐
│  alice_chain_id  (32B)                           │  ← 明文
├──────────────────────────────────────────────────┤
│  Enc(Bob.recv_pk, {                              │  ← Bob 的接收公钥加密
│    alice_ipv6           (16B)                    │
│    alice_port           (2B)                     │
│    beacon_nonce         (32B)  ← 原样回声        │
│    alice_ephemeral_pk   (32B)  ← 双棘轮临时公钥   │
│    timestamp            (8B)                     │
│  })                                              │
├──────────────────────────────────────────────────┤
│  Sign(Alice.send_sk, 上面全部)  (64B)            │  ← Alice 的发送私钥签名
└──────────────────────────────────────────────────┘
```

### 5.3 验证流程

```
Bob 收到回应包:
  ① 链上查 alice_chain_id → 获取 send_pk
  ② 验证 Sign(send_pk) → 失败丢弃 (防伪造)
  ③ 用自己的 recv_sk 解密 → 失败丢弃 (不是给我的)
  ④ 验证 beacon_nonce 匹配 → 不匹配丢弃 (防重放)
  ⑤ 验证 timestamp (5分钟窗口)
  ⑥ ✅ TCP 直连 alice_ipv6:alice_port → 双棘轮握手
```

### 5.4 安全属性

| 攻击 | 防御 |
|:---|:---|
| 读回应包内容 | 需要 Bob.recv_sk |
| 伪造回应包 | 需要 Alice.send_sk 签名 |
| 重放回应包 | beacon_nonce 已被 Bob 消费 |
| 篡改回应包 | Sign 失效 + GCM 认证失效 |

## 七、用户状态机

```
                      进程启动
                         │
                         ▼
              ┌────────────────────┐
              │ 尝试直连缓存好友 IP  │
              └──────┬──────┬──────┘
                     │      │
            连上一个以上   全部失败
                     │      │
                     ▼      ▼
             ┌──────────┐ ┌──────────────────┐
             │  ONLINE  │ │ 连接公开节点       │
             │ (快速上线)│ │ 发送匿名登陆信标    │
             └──────────┘ │ 状态: LIMBO       │
                          │ 此时:             │
                          │ · 不视为合法用户   │
                          │ · 网络图中不可见   │
                          │ · 不承载转发义务  │
                          └────────┬─────────┘
                                   │
                         ┌─────────┼─────────┐
                         │         │         │
                    好友解密回应  传播结束     │
                    反向连接     无人回应     │
                         │         │         │
                         ▼         ▼         │
                     ┌──────────┐ ┌────────┐ │
                     │  ONLINE  │ │ISOLATED│ │
                     │ 正式上线  │ │ 孤立   │ │
                     └──────────┘ └────────┘ │
                                            │
ISOLATED 状态:                               │
  · 好友 = ∅，无法通信                        │
  · 可向链上任意 ID 发信标 (但大概率被无视)     │
  · 等待好友上线 (好友会发信标找到自己)         │
  · 或通过已有在线好友获取新好友公钥            │
```

### 梯度重试策略

```
Limbo 状态下信标重试:
  第 0 次: 立即发送
  第 1 次: 5 秒后重发 (新 random_tag)
  第 2 次: 15 秒后重发
  第 3 次: 30 秒后重发
  
  任意时刻收到好友回应 → 立即取消后续重试 → ONLINE
  三轮全部无回应 → ISOLATED
```

## 八、节点体系

### 7.1 四类节点

| | 用户节点 (非正式) | 正式转发节点 | 正式记录节点 | 公开节点 |
|:---|:---|:---|:---|:---|
| **职责** | 通信 + 轻量转发 | 洋葱路由中转 | 存储完整链 | 接受匿名连接 |
| **信标转发** | ✅ | ✅ | ✅ | ✅ |
| **心跳扩散** | ✅ | - | - | - |
| **链存储** | - | - | ✅ | - |
| **工资** | 按单小费 | 资金池月结 | 资金池月结 | 无/自愿 |
| **准入** | 邀请即得 | 3月考核+考试 | 最高权重 | 公开注册 |
| **任期** | 无限 | 12月硬上限 | 12月硬上限 | 无限 |
| **担保责任** | 不参与 | 承担 | 承担 | 不承担 |
| **被协议信任** | ❌ | ❌ | ❌ | ❌ |

> 关键: 所有节点在协议层面均不被信任。信任仅存在于密码学层 (密钥持有者之间)。

### 7.2 公开节点

- **获取途径**: 邀请时附带 / DNS 种子 / 链上注册
- **职责**: 接受任意匿名 TCP 连接，转发 BEACON 帧
- **防护**: 速率限制 (每源 IPv6 每分钟 ≤ 5 信标) + 连接限额 + 信标防重放
- **转发策略**: 优先转发给自己的好友 → 好友不足时转发给其他公开节点 (混合策略)

### 7.3 节点升级路径

```
用户上线 → 非正式节点 (默认)
              │
              │ 3个月考核 + 能力考试 + 信任权重达标
              ▼
           正式节点 (任期 12 月)
              │
              │ 期满降级 (冷却 ≥ 3 月) / 末位淘汰
              ▼
           非正式节点 (回归预备役)
```

## 九、好友互联 · 信任图拓扑

### 8.1 核心原则

- **网络即社交图。** 边 = 线下邀请担保关系 = TCP 连接。
- **局部视图。** 每个节点只知自己的连接对象和好友的好友 (半径 ≤ 3)。
- **全局拓扑不存在于任何单一位置。**

### 8.2 路由表三环结构

```
第一环 (核心表):
  信任距离 = 1 (直接好友)
  TCP 持久连接 + 实时在线状态 + 双棘轮加密信道
  容量: 邓巴数上限 ~150

第二环 (邻居表):
  信任距离 ≤ 3
  通过心跳扩散获取 (一跳即止)
  存储: chain_id + 上次 IPv6 + 在线状态 + 通过谁可达
  容量: ~500-1000

第三环 (全局表):
  信任距离 > 3
  仅存 chain_id + 公钥哈希 (不存 IP)
  容量: 无上限 (链上可查)
```

### 8.3 心跳扩散

```
心跳包:
  { node_chain_id, epoch, neighbors: [{chain_id, ipv6, port, last_seen}, ...] }

扩散规则:
  · 仅发给信任距离 = 1 的好友 (一跳即止，不转发)
  · 频率: 活跃 30s / 空闲 5min
  · 邻居列表最多包含信任距离 ≤ 1 的节点
  
隐私保证:
  · 你的在线状态只有你的好友知道
  · 好友的好友知道你的存在性 (但不直接连接)
  · 三度之外完全不可知
```

### 8.4 社交图扩展

- **唯一合法扩展方式**: 通过现有在线好友介绍。
- 介绍人 (Alice) 同时是双方的好友 → 验证后传递公钥。
- 介绍人对被介绍人行为承担连带信誉责任。

## 十、防图分析措施

### 9.1 威胁

外部观察者通过分析 TCP 连接模式、流量时间关联、包大小等，可能推断社交关系。

### 9.2 防御层级

| 层级 | 措施 | 防御对象 | Phase |
|:---|:---|:---|:---|
| **连接面** | 固定连接预算 + 填充连接 | 外部被动观察者 | 1 |
| **流量面** | 固定包大小 + 恒定发送速率 | 外部被动观察者 | 1 |
| **时序面** | 随机延迟回应 (1-60s) + 随机延迟上线 (0-5min) | 外部被动观察者 | 1 |
| **路径面** | 洋葱路由 (≥3 跳) | 外部主动监控 | 2 |
| **信息论** | 全局图信息不存在于任何单一位置 | 内部攻击者 | 架构级 |

### 9.3 连接预算填充

```
每个节点维护固定连接数 (如 20):
  真实好友在线: N 人 → N 条真实连接
  填充连接: 20-N 条 → 随机连接公开节点或无关节点

填充连接:
  · 同样跑双棘轮 + 发送心跳
  · 载荷永远为空
  · 外部无法区分真实好友和填充
```

## 十一、消息路由

### 10.1 路由原则

- 消息永远沿信任边流动。
- 每跳转发者必须是发送方和接收方中至少一方的 (直接或间接) 好友。
- 无法本地路由时 → 触发信标发现。

### 10.2 路由查询

```
GetNextHop(target):
  ① target 在核心表中? → 直接发送
  ② target 在邻居表中? → 发给连接 target 的核心好友 (via)
  ③ 不在已知拓扑中 → 不可路由 → 触发信标发现
```

### 10.3 端到端安全

- 所有消息经双棘轮端到端加密。
- 转发节点只能看到路由头部 (下一跳目标)，看不到消息内容。
- 即使转发节点是好友本人，也无法读取发给第三方的内容。

## 十三、心跳扩散协议（详细设计）

### 12.1 包结构

| 字段 | 大小 | 说明 |
|:---|:---|:---|
| SenderChainID | 32B | 发送者链上ID |
| Epoch | 8B | 单调递增轮次号 |
| SenderIPv6 | 16B | 发送者当前地址 |
| SenderPort | 2B | 发送者监听端口 |
| Timestamp | 8B | 生成时间 |
| Neighbors | N×59B | 邻居快照列表 (最多150) |
| Signature | 64B | Ed25519(send_sk) |

每个 NeighborSnapshot:
| 字段 | 大小 | 说明 |
|:---|:---|:---|
| ChainID | 32B | 邻居链上ID |
| TrustDist | 1B | 始终=1 (对发送者而言) |
| LastSeen | 8B | 发送者上次看到的时间戳 |
| IPv6 | 16B | 邻居IPv6 |
| Port | 2B | 邻居端口 |

### 12.2 活跃/空闲判定

- 活跃: 过去 2 分钟内有收发 DATA 帧 → 心跳间隔 30s ± 5s
- 空闲: 无 DATA 帧超过 2 分钟 → 心跳间隔 300s ± 30s

### 12.3 接收处理

```
收到心跳 → 验证send_pk签名 → 更新核心表(sender在线) →
遍历Neighbors → 不在核心表则加入邻居表(trustDist=2, via=sender) →
清理超过15分钟未更新的邻居条目
```

### 12.4 关键约束

- **一跳即止**: 收到的心跳不转发
- **邻居列表只含距离=1**: 不泄露「好友的好友」给好友
- **签名绑定**: 防止中转节点篡改邻居列表

## 十四、连接预算填充（详细设计）

### 13.1 核心参数

- 固定连接数: 20 (可配置)
- 维护周期: 每 60 秒检查一次
- 真实好友连接: 始终保留，不受预算驱逐

### 13.2 填充目标选择

优先级:
1. 公开节点 (始终在线)
2. 邻居表中标记在线的节点
3. 全局已知节点 (任意)

### 13.3 填充连接行为

```
与真实连接完全相同:
  · TCP + TLS/Noise 握手
  · 双棘轮初始化
  · 定期心跳 (载荷为空)
  · 固定大小填充包 (512B)

唯一区别: 握手时交换一条加密标记 "此为填充连接"
双方知悉 → 双方发空载荷 → 可随时静默断开
```

### 13.4 安全性

- 外部观察者: 始终看到 20 条加密连接，无法区分真实与填充
- 好友数变化: 不影响对外连接数
- 好友增减: 只改变填充连接数，不改变总连接数

## 十五、随机延迟参数（详细设计）

### 14.1 回应延迟

- 场景: Alice 解密信标 → 发送回应包
- 分布: 指数分布，均值 15s，截断到 [1s, 60s]
- 效果: 大多聚集在 15s 附近，少量更长，自然分布

### 14.2 上线延迟

- 场景: 程序启动 → 首次发送信标
- 分布: 均匀分布 [0, 5分钟]
- 效果: 去耦合「程序启动时间」与「网络活动时间」

### 14.3 转发抖动

- 场景: 中转节点转发信标
- 分布: 均匀 [10ms, 500ms]
- 效果: 打破跨跳时序关联

### 14.4 心跳抖动

- 活跃: 30s ± 5s
- 空闲: 300s ± 30s
- 效果: 避免所有节点同时心跳

## 十六、公开节点 DNS 种子（详细设计）

### 15.1 DNS 记录

- 种子域名: `seed-{N}.v6.neko.network` (N=1..10, 客户端硬编码)
- AAAA: 返回公开节点 IPv6 地址列表
- TXT: 返回版本信息 (`v=1; min-proto=1; network=neko-main`)

### 15.2 查询流程

```
并发查所有种子域名 → 汇总IPv6列表 → 去重 → 随机打乱 → 逐一尝试连接
全部失败 → 回退到上次缓存 → 缓存也空 → 无网络连接
```

### 15.3 身份验证

连接公开节点后:
1. TLS/Noise 握手
2. 对方发送 chain_id
3. 链上查询: 该 chain_id 是否注册为公开节点?
4. 验证通过 → 接受; 失败 → 断开, 试下一个

## 十七、好友介绍协议（详细设计）

### 16.1 前提

- Bob 和 Alice 是在线好友 (trustDist=1)
- Alice 和 Charlie 是在线好友
- Bob 没有 Charlie 的公钥

### 16.2 完整流程

```
① Bob → Alice (加密信道): "我想认识 Charlie (chain_id=0xCHARLIE)"
② Alice 验证: Charlie 是我的好友? 我担保了 Charlie?
③ Alice 构造介绍凭证 + 签名
④ Alice → Bob: Charlie的公钥 + 介绍凭证
⑤ Bob 向 Charlie 发信标 (载荷含介绍凭证)
⑥ Charlie 解密 → 验证Alice签名 → Alice是我的好友? → 高概率回应
⑦ Bob ↔ Charlie 建立连接, trustDist=2
```

### 16.3 介绍凭证结构

| 字段 | 大小 | 说明 |
|:---|:---|:---|
| Introducer | 32B | Alice 的 chain_id |
| SubjectA | 32B | Bob 的 chain_id |
| SubjectB | 32B | Charlie 的 chain_id |
| Timestamp | 8B | 签发时间 |
| Expiry | 8B | 签发时间 + 30天 |
| Signature | 64B | Alice.send_sk 签名 |
| **总计** | **176B** | 作为信标载荷扩展字段 |

### 16.4 Charlie 的决策

```
① 验证 Alice 签名 → 失败丢弃
② Alice 是我的好友? → 否 → 丢弃
③ 凭证过期? → 是 → 丢弃
④ Alice 信任权重足够? → 否 → 丢弃
⑤ ✅ 全部通过 → 标记为「介绍来的」→ 高优先级回应
```

### 16.5 信任影响

- Alice 的信誉与此次介绍绑定
- Bob 或 Charlie 恶意行为 → Alice 信任权重连带受损
- 凭证 30 天过期 → 过期后双方需重新通过直接通信维持关系

## 十八、双棘轮端到端加密（详细设计）

### 17.1 设计目标

| 属性 | 机制 | 效果 |
|:---|:---|:---|
| **前向安全** | 消息密钥用完即毁 | 当前密钥泄露 → 历史消息安全 |
| **后向安全** | DH棘轮每次混入新共享密钥 | 当前密钥泄露 → 未来消息自愈 |
| **防篡改** | AES-GCM认证标签 | 任何篡改 → 解密失败 |
| **防重放** | MsgNum单调递增 + SkippedMsgKeys去重 | 重放消息 → 解密但无效 |
| **乱序容忍** | SkippedMsgKeys存储跳过的消息密钥 | 乱序到达 → 查找对应密钥 |

### 17.2 核心结构

双棘轮 = **DH棘轮**（外层，后向安全）+ **对称棘轮**（内层，前向安全）

```
DH棘轮:
  每次收到对方新DH公钥 → 混入当前RootKey
  → 新RootKey + 新接收链密钥

对称棘轮:
  链密钥 → HMAC(链密钥, 0x02) = 消息密钥 → 用完即毁
  链密钥 → HMAC(链密钥, 0x01) = 新链密钥
```

### 17.3 状态结构

```
RatchetState:
  DHs           [32]byte  我的当前DH私钥
  DHr           [32]byte  对方的当前DH公钥
  RootKey       [32]byte  根密钥（每次DH棘轮更新）
  SendChainKey  [32]byte  当前发送链密钥
  SendMsgNum    uint32    已发送消息数
  RecvChainKey  [32]byte  当前接收链密钥
  RecvMsgNum    uint32    已接收消息数
  SkippedMsgKeys map[[8]byte][32]byte  乱序消息密钥缓存
  MaxSkip       uint32    最多跳过消息数（防DoS，默认1000）
  PrevSendChainKey [32]byte  前一条发送链密钥
  PrevSendMsgNum   uint32    前一条发送链消息总数
```

### 17.4 密钥推导函数

```
KDF_RK(rootKey, dhOutput) → (newRootKey, newChainKey):
  prk = HMAC-SHA256(rootKey, dhOutput)
  newRootKey  = HMAC-SHA256(prk, 0x01)
  newChainKey = HMAC-SHA256(prk, 0x02)

KDF_CK(chainKey) → (newChainKey, messageKey):
  newChainKey = HMAC-SHA256(chainKey, 0x01)
  messageKey  = HMAC-SHA256(chainKey, 0x02)
```

### 17.5 初始化

```
InitAsInitiator(bobIdentitySK, aliceIdentityPK, aliceEphemeralPK):
  ① 生成 bobEphemeralSK, bobEphemeralPK
  ② dh1 = DH(bobEphemeralSK, aliceEphemeralPK)
     dh2 = DH(bobIdentitySK, aliceIdentityPK)
  ③ RootKey = SHA256(dh1 || dh2)
  ④ DHs = bobEphemeralSK, DHr = aliceEphemeralPK
  ⑤ 返回 bobEphemeralPK（需要发给Alice）

InitAsResponder(aliceIdentitySK, bobIdentityPK, aliceEphemeralSK, bobEphemeralPK):
  ① dh1 = DH(aliceEphemeralSK, bobEphemeralPK)
     dh2 = DH(aliceIdentitySK, bobIdentityPK)
  ② RootKey = SHA256(dh1 || dh2)
  ③ DHs = aliceEphemeralSK, DHr = bobEphemeralPK
```

### 17.6 加密流程

```
Encrypt(plaintext):
  ① 保存当前发送链: PrevSendChainKey, PrevSendMsgNum
  ② 生成新DH对: newDHSK, newDHPK
  ③ DH棘轮: dhOutput = DH(newDHSK, DHr)
     RootKey, SendChainKey = KDF_RK(RootKey, dhOutput)
  ④ DHs = newDHSK, SendMsgNum = 0
  ⑤ 对称棘轮: SendChainKey, messageKey = KDF_CK(SendChainKey)
  ⑥ Header = { newDHPK, PrevSendMsgNum, SendMsgNum }
  ⑦ ciphertext = AES-256-GCM(messageKey, plaintext, Header)
  ⑧ 销毁 messageKey → 返回 Header || ciphertext
```

### 17.7 解密流程

```
Decrypt(wireMessage):
  ① 解析 Header { DHPublicKey, PrevChainCount, MsgNum }
  ② 如果 DHPublicKey != DHr:
       → 保存跳过消息: skipMessageKeys(PrevChainCount)
       → DH棘轮: dhRatchetStep(DHPublicKey)
  ③ 跳过已处理: skipMessageKeys(MsgNum)
  ④ RecvChainKey, messageKey = KDF_CK(RecvChainKey)
  ⑤ plaintext = AES-256-GCM_Decrypt(messageKey, ciphertext, Header)
  ⑥ 销毁 messageKey → RecvMsgNum++ → 返回 plaintext

dhRatchetStep(newDHPublicKey):
  dhOutput = DH(DHs, newDHPublicKey)
  RootKey, RecvChainKey = KDF_RK(RootKey, dhOutput)
  DHr = newDHPublicKey, RecvMsgNum = 0
```

### 17.8 乱序处理

```
skipMessageKeys(until):
  while RecvMsgNum < until:
    RecvChainKey, mk = KDF_CK(RecvChainKey)
    SkippedMsgKeys[key(DHr, RecvMsgNum)] = mk
    RecvMsgNum++

解密时:
  如果 MsgNum < RecvMsgNum:
    从 SkippedMsgKeys 查找对应消息密钥
    解密后删除该条目
```

### 17.9 线上消息格式

```
[Header: 40B || AES-GCM Ciphertext]

Header:
  DHPublicKey     [32]byte  发送方当前DH公钥
  PrevChainCount  uint32    前一条发送链的消息总数
  MsgNum          uint32    当前链中的消息序号

Ciphertext:
  AES-256-GCM 密文 + 16B 认证标签
```
