## 一、技术栈决策

### 1.1 选型

| 组件 | 选型 | 版本 | 用途 |
|:---|:---|:---|:---|
| 链框架 | Cosmos SDK | v0.50+ | 模块化区块链框架 |
| 共识 | Tendermint (CometBFT) | v0.38+ | BFT共识引擎 |
| 状态存储 | IAVL + BadgerDB | 内置 | Merkle状态树 |
| ZK框架 | gnark | v0.9+ | Groth16电路 |
| ZK曲线 | BN254 / BLS12-381 | gnark-crypto | 配对友好曲线 |
| 签名 | Ed25519 | 标准库 | 交易签名 |
| KEM | ECDH (Curve25519) | 标准库 | 公钥加密 |
| 序列化 | Protobuf | Cosmos内置 | 链上数据编码 |

### 1.2 不选用的理由

| 不选 | 理由 |
|:---|:---|
| 自建链 | 重复造共识/存储/RPC，工作量 ×10 |
| go-ethereum | 单体架构，模块化差，EVM与本项目需求错配 |
| Risc0 / SP1 | RUST ZK方案，与Go技术栈不统一 |
| PLONK | 证明体积大于Groth16，验证时间更长 |

### 2.1 架构定位

**`neko-node` 是中间层守护进程，不是面向用户的应用程序。**

```
┌──────────────────────────────────────────────────────────────┐
│                    用户界面层 (独立进程)                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐             │
│  │ 聊天应用  │ │ 交易应用  │ │ 游戏客户端        │             │
│  └────┬─────┘ └────┬─────┘ └────────┬─────────┘             │
│       │            │               │                         │
│       └────────────┼───────────────┘                         │
│                    │ gRPC + WebSocket (localhost)            │
├────────────────────┼─────────────────────────────────────────┤
│              ┌─────┴─────┐                                    │
│              │ neko-node │  ← 守护进程 (本文档范围)            │
│              │ (中间层)   │   开机自启，常驻后台                │
│              └───────────┘                                    │
├──────────────────────────────────────────────────────────────┤
│                    IPv6 P2P · 其他节点                        │
└──────────────────────────────────────────────────────────────┘
```

上层应用（聊天/交易/游戏）通过 `localhost` 的 gRPC 和 WebSocket 与 `neko-node` 通信。用户看不到 `neko-node`——用户看到的是聊天界面、商城页面、游戏画面。

### 2.2 本地 API 接口设计

```protobuf

    // === 事件订阅 ===

    // → 获取分布式随机信标当前轮次结果
    rpc QueryGameState(QueryGameStateRequest) returns (GameState);
    // → 查询游戏链上状态
    
    // === 节点管理 ===
    rpc GetNodeStatus(Empty) returns (NodeStatus);
    // → { sync_profile, peers_online, chain_height, pool_balance }
    rpc SetRole(SetRoleRequest) returns (SetRoleResponse);
    // → 申请角色切换 (用户主动触发，需满足条件)
```

## 三、明链模块 (`x/brightchain`)

### 3.1 模块职责

- 用户注册（双密钥锚定到 chain_id）
- 还款记录（语义翻转：向资金池放贷）
- 资金池管理（余额追踪、支出审批）
- 资金质押担保（公开担保关系 + 信用票据质押）
- 递归追偿（质押金优先赔付）
- 信任权重动态计算

### 3.2 自定义账户类型 (UserBlock)

**每个用户拥有一个属于自己的链上数据块，只能被自己更新。用户只需保留双密钥即可重建一切。**

```protobuf
message BrightAccount {
    // === 继承 Cosmos BaseAccount ===
    string   address    = 1;  // chain_id 的 bech32 编码
    uint64   account_number = 2;
    uint64   sequence   = 3;  // 交易序号 (防重放)
    
    // === 双密钥 ===
    bytes    recv_pk    = 4;  // 32B: Curve25519 公钥 (别人加密给我)
    bytes    send_pk    = 5;  // 32B: Ed25519 公钥 (验证我的签名)
    
    // === 信誉系统 ===
    uint64   credit_score       = 6;
    uint64   trust_weight       = 7;
    uint64   total_repay_amount = 8;
    uint64   credit_limit       = 9;
    
    // === 种子期标记 ===
    bool     seed_phase         = 10;
    repeated bytes guarantors   = 11;  // 担保人 chain_id 列表
    
    // === 节点角色 ===
    NodeRole node_role          = 12;
    int64    node_term_start    = 13;
    int64    node_term_end      = 14;
    
    // === 好友列表 (链上存储，用户签名更新) ===
    repeated FriendRecord friends = 15;
    // 换设备 → 导入密钥 → 查自己的 UserBlock → 好友列表恢复
    
    // === 担保关系 (指到自己 UserBlock 里的好友) ===
    repeated BondRef guaranteed_of  = 16;  // 我担保了谁
    repeated BondRef guaranteed_by  = 17;  // 谁担保了我
}

message FriendRecord {
    bytes    chain_id      = 1;
    bytes    recv_pk       = 2;  // 好友的接收公钥
    bytes    send_pk       = 3;  // 好友的发送公钥
    uint32   trust_dist    = 4;  // 1=直接好友 2=好友介绍
    int64    added_at      = 5;
    bytes    introduced_by = 6;  // 介绍人 chain_id (trust_dist=2时)
}

message BondRef {
    string   bond_id       = 1;
    bytes    other_party   = 2;  // 对方 chain_id
}
```

### 3.3 存储模型

```
链上 (IAVL 树):
  UserBlock: chain_id → { 密钥 + 好友 + 担保 + 信用 }
  BondRegistry: bond_id → { 质押 + 状态 }
  PoolBalance: uint64
  CreditTree: Merkle(信用票据)

本地 (~/.nekop2p/):
  keys.json    ← 双密钥 (唯一必须保留的文件)
  peer_cache/  ← 可选: IPv6/端口 心跳缓存

不需存储:
  ❌ 好友列表 → 链上 UserBlock
  ❌ 交易历史 → 链上查询
  ❌ 担保关系 → 链上查询
```

### 3.6 担保质押系统 (GuaranteeBond)

**原则: 担保 = 信用票据质押，而非口头承诺。**

```
流程:
  A邀请B → A质押信用票据作为担保金 → B获得种子信用额度 = 质押额×系数
  B违约 → A的质押金被没收赔付 → A损失真金白银
  B还清 → A的质押金解冻释放
```

#### 质押状态

```protobuf
message GuaranteeBond {
    string   bond_id         = 1;
    string   inviter         = 2;  // 担保人 chain_id (公开)
    string   invitee         = 3;  // 被担保人 chain_id (公开)
    repeated CreditNoteID locked_notes = 4;  // 被锁定的信用票据
    uint64   total_bond      = 5;  // 质押总额
    uint64   coefficient     = 6;  // ≤ 1.0
    uint64   seed_limit      = 7;  // B的种子信用额度
    int64    locked_at       = 8;
    int64    unlock_at       = 9;  // 或提前释放
    BondStatus status        = 10; // ACTIVE | RELEASED | FORFEITED
}
```

#### 多担保人

```
B需要 ≥3 个担保人，各自独立质押

A1质押500 → B额度400, A2质押300 → B额度240
A3质押200 → B额度160
B总种子额度 = 800

B违约600 → 按质押比例扣除各担保人质押金:
  A1损失300, A2损失180, A3损失120
```

#### 递归追偿更新

```
债务追偿顺序:
  ① B自身余额
  ② 担保人质押金 (优先赔付)
  ③ 各担保人自身余额
  ④ 担保人的担保人 (递归上溯)
  ⑤ 坏账准备金兜底
```

#### 隐私模型

```
chain_id = 网名。担保关系在明域公开，但不关联真实身份。

链上可见: 0xAAAA 质押1000担保了 0xBBBB
链上不可见: 0xAAAA 是谁、0xBBBB 是谁
关联只在: 线下认识的人脑子里
模型: 与比特币地址假名一致
```

## 四、暗链模块 (`x/darkchain`)

### 4.1 模块职责

- 借贷申请与审批
- 债务状态追踪
- 匿名化名管理
- 信用票据系统 (防无限化名生成)
- 交易结算指令生成

### 4.2 信用票据系统 (Credit UTXO)

**问题：** 匿名化名可无限生成。恶意用户可用不同化名反复借贷并违约。信用必须是**一次性可消费的票据**，而非可复用的静态证明。

**方案：** 借鉴 Zcash UTXO 模型——信用以「票据」形式存在，每张票据只能花费一次。

#### 票据结构

```protobuf
message CreditNote {
    uint64   value       = 1;  // 面额
    bytes    owner_key   = 2;  // 由 master_secret 派生 (不暴露链上身份)
    bytes    serial      = 3;  // 唯一序列号 (随机)
    bytes    commitment  = 4;  // Hash(value, owner_key, serial)
}
```

#### 票据生命周期

```
创建:
  · 初始分配: 种子用户获得初始信用票据
  · 还款: 等额新票据 (还多少生成多少)
  · 赚取: 明域活动/交易赚取信用 → 新票据

花费 (借贷):
  · 选择票据作为借款的信用背书
  · 每张票据 → 生成 nullifier = PRF(master_secret, serial)
  · nullifier 记录到暗链全局 nullifier 集合
  · 票据被消费，不可再使用

销毁:
  · 违约: 票据已被消费 (nullifier已记录)，信用不可恢复
  · 过期: 长期未使用票据可设有效期
```

#### 防无限化名攻击

```
Alice (credit_score=5000, 持有票据A,B,C 各500)

① anon_id_1 → 花费票据A → nullifier_A 记录 → 借500 → 违约
② anon_id_2 → 花费票据B → nullifier_B 记录 → 借500 → 违约
③ anon_id_3 → 需要票据 → 票据C (价值500)
④ anon_id_4 → 没有未花费票据了 → zk证明失败 → 无法借贷

跑路 = 烧票据 = 信用不可复用
新票据只能通过: 还债 / 明域赚取 / 他人转让
```

#### 全局状态

```
明链信用树 (Merkle Tree):
  root = Merkle(所有未花费票据的 commitment)
  
暗链 nullifier 集合:
  { nullifier_1, nullifier_2, ... }
  同一 serial → 同一 nullifier → 第二次花费被拒绝
```

#### 借贷时的 ZK 证明扩展

```
MsgRequestLoan 的 zk_credit_proof 现在包含:

输入票据:
  for each input_note:
    ① commitment 在明链信用树中 (Merkle证明)
    ② owner_key = DerivePublic(master_secret, note_index)
    ③ nullifier = PRF(master_secret, serial) — 存入暗链nullifier集合
    ④ nullifier 不在已花费集合中

输出票据 (找零):
  输入总额 ≥ 借款额 + 输出总额
  输出票据的 commitment 加入信用树 (新票据)

隐私保证:
  外部可见: "有N张票据被消费，M张新票据被创建"
  不可见: 票据属于谁、来自哪个明链账户
```

### 4.3 数据结构

```protobuf
// === 匿名化名 (与明链身份分离) ===
message AnonIdentity {
    bytes    anon_id       = 1;  // PRF(master_secret, counter)
    bytes    zk_credit_proof = 2; // ZK信用证明 (不暴露明链身份)
    int64    created_at    = 3;
    int64    expires_at    = 4;  // 化名有效期
}

// === 借条 (暗链核心记录) ===
message LoanRecord {
    string   loan_id       = 1;  // 唯一借条ID
    bytes    borrower_anon = 2;  // 借款人匿名化名
    bytes    lender_anon   = 3;  // 贷款人匿名化名 (P2P) 或 资金池
    uint64   amount        = 4;  // 金额
    int64    term_days     = 5;  // 期限 (天)
    LoanStatus status      = 6;  // PENDING | ACTIVE | SETTLED | DEFAULTED
    
    bytes    zk_credit_proof = 7;  // 借款人信用证明
    bytes    inkwell_params  = 8;  // 混沌结算参数
    
    int64    created_at    = 9;
    int64    due_date      = 10;
    int64    settled_at    = 11;
    
    // 还款碎片追踪
    repeated RepayFragment fragments = 12;
}

message RepayFragment {
    uint64   amount;
    string   payer_anon;     // 付款人匿名化名 (可能不是借款人本人)
    int64    paid_at;
    bytes    zk_repay_proof; // 此碎片的还款证明
}

enum LoanStatus {
    PENDING   = 0;  // 审批中
    ACTIVE    = 1;  // 已放款，还款中
    SETTLED   = 2;  // 结清
    DEFAULTED = 3;  // 违约
}
```

### 4.5 匿名化名生成

```go
// PRF: 伪随机函数，生成无法追溯到明链身份的化名
func GenerateAnonID(masterSecret [32]byte, counter uint64) [32]byte {
    // anon_id = BLAKE3(master_secret || counter)
    input := make([]byte, 40)
    copy(input[:32], masterSecret[:])
    binary.BigEndian.PutUint64(input[32:], counter)
    return blake3.Sum256(input)
}

// master_secret: 用户本地生成，永不上链
// counter: 每次交易递增
// 结果: 外部无法关联 anon_id → chain_id
//       用户自己可以随时证明 "这个anon_id是我的" (提供master_secret证明)
```

### 4.6 门锁机制 (Identity Marker)

**问题:** 同一用户可生成多个匿名化名，用化名A卖给化名B制造虚假交易刷信用。

**方案:** 每个暗链区块生成周期标记，每个用户必须提交基于该标记的身份标记。同一区块内相同标记 = 同一个人 = 拒绝。

#### 周期标记

```go
// 每个暗链区块自动生成
cycle_marker = BLAKE3(dark_chain_prev_block_hash, block_height)

// 每个用户对该周期生成唯一身份标记
identity_marker = PRF(master_secret, cycle_marker)
```

#### 验证逻辑

```
暗链交易提交时附带:
  { identity_marker, zk_proof(正确派生) }

验证器:
  ① zk_proof: identity_marker = PRF(master_secret, cycle_marker) ✅
  ② identity_marker ∈ 已使用标记集合? 
     → 是: 拒绝 (同一人用不同化名在同一区块双重出现)
     → 否: 接受 → 标记加入集合 → 区块结束时清空
```

#### 属性

| | 效果 |
|:---|:---|
| 同人同区块双化名 | 碰撞 → 拒绝 ✅ |
| 同人不同区块 | cycle_marker不同 → 标记不同 → 允许 ✅ |
| 外部追踪 | PRF单向 → 无法从标记反推master_secret ✅ |
| 跨区块关联 | 不同周期不同标记 → 无法关联 ✅ |

#### ZK电路

```go
type IdentityMarkerCircuit struct {
    // 公开输入 (~500个约束，验证时间~2ms)
    CycleMarker     frontend.Variable `gnark:",public"`
    IdentityMarker  frontend.Variable `gnark:",public"`
    
    // 秘密输入
    MasterSecret    frontend.Variable
    
    // 约束: identity_marker = PRF(master_secret, cycle_marker)
}
```

#### 与信用票据的关系

```
信用票据: 管「信用能不能花」(一张票据只能花一次)
门锁:     管「人能不能出现多次」(一人一区块只能出现一次)

两个机制互补，覆盖暗域全部欺诈路径:
  女巫刷单    → 门锁拒绝
  超额借贷   → 信用票据限制
  自交伪造   → 门锁拒绝
  票据双花   → nullifier集合拒绝
```

### 5.3 参数生成算法

```go
// 在借贷审批时，双方共同调用此算法生成混沌参数
func GenerateInkwellParams(loanID string, seed [32]byte, totalAmount uint64) InkwellParams {
    rng := newPRNG(seed)  // 确定性随机数生成器
    
    // === 锁1: 时间 ===
    windowStart := time.Now().Add(24 * time.Hour).Unix()
    windowEnd   := windowStart + int64(rng.Intn(90*24*3600)) // 1~90天
    fragCount   := 3 + rng.Intn(5)  // 3~7片
    
    // === 锁2: 金额 ===
    fragments := splitAmount(totalAmount, fragCount, rng)
    
    // === 锁3: 来源 ===
    relayEnabled := rng.Float64() < 0.3  // 30%概率启用接力
    relayPath := selectRelayNodes(rng)
    
    return InkwellParams{
        LoanID:        loanID,
        Seed:          seed,
        TotalAmount:   totalAmount,
        WindowStart:   windowStart,
        WindowEnd:     windowEnd,
        FragmentCount: uint32(fragCount),
        Fragments:     fragments,
        RelayEnabled:  relayEnabled,
        RelayPath:     relayPath,
    }
}

// 金额拆封: 保证总和 = totalAmount，碎片大小随机但不为零
func splitAmount(total uint64, count int, rng *PRNG) []uint64 {
    // Dirichlet分布: 随机[0,1] → 归一化 → 乘以总额
    weights := make([]float64, count)
    sum := 0.0
    for i := range weights {
        weights[i] = rng.Float64() + 0.1  // +0.1 防止零碎片
        sum += weights[i]
    }
    
    fragments := make([]uint64, count)
    allocated := uint64(0)
    for i := 0; i < count-1; i++ {
        fragments[i] = uint64(float64(total) * weights[i] / sum)
        allocated += fragments[i]
    }
    fragments[count-1] = total - allocated  // 最后一块补足差值
    return fragments
}
```

### 5.4 还款时间调度

```go
// 每次还款时调用，决定本次应还哪个碎片
func (k Keeper) GetNextRepayFragment(loan LoanRecord, currentTime int64) (*RepayFragment, error) {
    params := loan.InkwellParams
    
    // 根据时间和种子确定当前应还的碎片索引
    rng := newPRNG(concat(params.Seed, int64ToBytes(currentTime)))
    idx := rng.Intn(len(params.Fragments))
    
    // 检查该碎片是否已还
    if alreadyPaid(loan, idx) {
        // 找下一个未还的碎片
        for offset := 0; offset < len(params.Fragments); offset++ {
            nextIdx := (idx + offset) % len(params.Fragments)
            if !alreadyPaid(loan, nextIdx) {
                return &RepayFragment{
                    Amount:   params.Fragments[nextIdx],
                    FragmentIndex: nextIdx,
                    DueTime:  currentTime + 24*3600, // 24小时内还款
                }, nil
            }
        }
        return nil, ErrAllFragmentsPaid
    }
    
    return &RepayFragment{
        Amount:       params.Fragments[idx],
        FragmentIndex: idx,
        DueTime:      currentTime + 24*3600,
    }, nil
}
```

### 6.3 身份证明电路 (最复杂的一组)

```go
// identity/circuit.go

type IdentityCircuit struct {
    // === 公开输入 (链上任何人都能看到) ===
    MySendPK        frontend.Variable  `gnark:",public"`  // 我的send_pk
    
    // === 秘密输入 (zk保护) ===
    MyRecvPK        frontend.Variable  // 我的recv_pk
    Inviters        [3]InviterWitness  // 担保人信息
    MerkleProof     MerkleProof        // 担保人在链上注册的默克尔证明
}

type InviterWitness struct {
    SendPK         frontend.Variable  // 担保人send_pk
    Signature      frontend.Variable  // 担保人的Ed25519签名
    SignedMessage  frontend.Variable  // 被签名消息 (包含my_send_pk)
    TrustWeight    frontend.Variable  // 担保人信任权重
}

// 约束:
// ① 每个担保人的 Ed25519 签名验证通过
// ② 被签名的消息中包含 MySendPK (担保人确认担保了我)
// ③ 担保人的 send_pk 在默克尔树中 (担保人已在链上注册)
// ④ 担保人不在黑名单中
// ⑤ 至少3个担保人满足信任权重 > 阈值

func (c *IdentityCircuit) Define(api frontend.API) error {
    var validCount frontend.Variable = 0
    
    for _, inviter := range c.Inviters {
        // ① Ed25519 签名验证 (gnark 标准库)
        ed25519.Verify(api, inviter.Signature, inviter.SignedMessage, inviter.SendPK)
        
        // ② 验证签名的消息包含 MySendPK
        // signedMessage = SHA256("invite:" || MySendPK || timestamp)
        // 简化为: 验证最后32字节等于 MySendPK
        verifyContainsPK(api, inviter.SignedMessage, c.MySendPK)
        
        // ③ 担保人在默克尔树中
        verifyMerkleProof(api, c.MerkleProof.Root, c.MerkleProof.Path, inviter.SendPK)
        
        // ④ 担保人信任权重 > 阈值
        weightOk := api.Cmp(inviter.TrustWeight, MinTrustWeight)
        validCount = api.Add(validCount, weightOk)
    }
    
    // ⑤ 有效担保人数 ≥ 3
    api.AssertIsLessOrEqual(3, validCount)
    
    return nil
}
```

```go
func (k Keeper) VerifyIdentityProof(ctx sdk.Context, proof []byte, publicInput MySendPK) error {
    // ① 获取验证密钥
    vk := k.GetVerifyingKey(ctx, "identity")
    
    // ② 反序列化证明

    assignment := []frontend.Variable{publicInput}
```

## 十、节点同步架构

### 10.1 统一节点模型

**一个二进制 (`neko-node`)，N种角色。** 不区分客户端/服务器程序。角色由同步深度 + 义务集合决定。

### 10.2 三种同步深度

| | 轻量同步 | 标准同步 | 完整同步 |
|:---|:---|:---|:---|
| **适用角色** | ISOLATED / 匿名登陆 | 用户节点 / 非正式节点 / 公开节点 | 正式记录节点 / 正式转发节点 |
| **区块头** | ✅ | ✅ | ✅ |
| **自身账户状态** | ✅ | ✅ | ✅ |
| **好友公钥 (距离≤3)** | — | ✅ | ✅ |
| **好友账户状态** | — | ✅ | ✅ |
| **公开节点列表** | ✅ | ✅ | ✅ |
| **全局注册表** | — | — | ✅ |
| **完整交易历史** | — | — | ✅ |
| **ZK验证密钥** | ✅ | ✅ | ✅ |
| **存储量估算** | ~50MB | ~500MB | ~50GB+ |
| **转发义务** | — | ✅ | ✅ |
| **存储义务** | — | — | ✅ |
| **工资** | — | 按单小费 | 月结 |

### 10.3 角色迁移与同步深度切换

```
ISOLATED ──→ 轻量同步
     │ (好友上线)
     ▼
用户节点 ──→ 标准同步
     │ (考核通过)
     ▼
正式节点 ──→ 完整同步
     │ (任期到期/淘汰)
     ▼
用户节点 ──→ 标准同步 (可清理非必要数据)
```

### 10.4 同步方式 (Tendermint 内置)

- **快速同步**: 新区块头 + 状态快照 → 分钟级加入
- **WebSocket 订阅**: 实时接收新区块事件
- **ABCI Query**: 按需查询特定账户/公钥，不下载全量

> Cosmos SDK 标准功能，无需额外实现。
