## 二、设计哲学

### 2.1 第一性原理

**可验证性取代信任。** 每一步操作通过链上数据、零知识证明或公开算法实现事后可审计、可重放、可独立验证。信任权威被替换为验证机制。

### 2.2 双层城邦模型

| 层级 | 别名 | 特征 | 目标 |
|:---|:---|:---|:---|
| **地面层** | 阳光市场 | 透明、实名信用、高速 | 日常商业活动 |
| **地下层** | 防空洞 | 匿名、零知识遮蔽、慢速 | 紧急生存通道 |
| **楼梯** | 切换机制 | 物理拨片、ZK身份剥离 | 两层间可控跳转 |

市场养防空洞：地面交易手续费抽成固定划拨给地下层矿工维护费。每一个交易者都是全网「紧急生存通道」的投保人。

### 2.3 核心原则

1. **经济闭环**: 交易量 (GDP) 决定节点数。网络维护成本由内部经济活动支付。
2. **权力显式化**: 所有权威角色的权力边界明确、操作链上留痕、重大调整有公告期。
3. **节点即非信任实体**: 协议层不信任任何转发/存储节点。信任仅存在于密码学密钥持有者之间。
4. **局部视图，全局不可知**: 每个节点只知半径 ≤ 3 的拓扑。全局图不存在于任何单一位置。

### 2.4 身份隐私模型

```
三层身份，两层密码学隔离:

  真实身份 (张三)
     ↕ 仅存于线下人际关系的记忆中，系统不收集
  ┌──────────────────────────────────────┐
  │ bright_keys.json (用户保管)          │
  │   recv_sk + send_sk                 │
  │   → chain_id (0xABCD...) = 网名     │
  │   → 明域: 公开活动、担保、信用       │
  ├──────────────────────────────────────┤
  │ dark_keys.json (用户保管)            │
  │   master_secret (32B随机)           │
  │   → 匿名化名 · identity_marker     │
  │   → 暗域: 匿名借贷、混沌结算         │
  └──────────────────────────────────────┘

密码学保证:
  持有 bright_keys → 可以追查明域所有活动
                   → 不能追查暗域活动 (anon_id = PRF(master_secret, ...))
                   → 不能关联暗域化名到明域 chain_id
                   
  持有 dark_keys  → 可以花费匿名信用票据
                  → 不能反向推导 chain_id
                  → ZK证明是单向桥接: 只证明「我有信用」不暴露「我是谁」

两个密钥文件完全独立。丢失任一个 = 丢失对应域的身份。
```

### 2.5 担保模型：资金质押

担保 = 信用票据质押，而非口头承诺。每次担保都是真金白银的风险评估。

```
A质押票据 → B获得种子信用额度(≤质押额)
B违约 → A质押金被没收赔付
B还清 → A质押金解冻

递归追偿: 质押金优先赔付 → 担保人余额 → 递归上溯 → 坏账准备金
```

## 三、物理层设计

### 3.1 网络约束

- **仅 IPv6。** 每节点全球可路由地址，无需 NAT 穿透。
- 传输层: TCP (Phase 1)，后续升级 QUIC。

### 3.2 双密钥载体

每个用户在链上持有两个完全无关的密钥对：

```
链上存储 (chain_id →):
  recv_pk    ← 别人用来加密发给我的消息 (只有我能解)
  send_pk    ← 别人用来验证是我发的消息 (只有我能签)
  zk_proof   ← 我是被邀请的真实人类

本地持有:
  recv_sk    ← 解密别人发给我的消息
  send_sk    ← 签名我发出的消息
```

| 密钥对 | 公开 | 私持 | 用途 |
|:---|:---|:---|:---|
| recv | recv_pk | recv_sk | 任何人→我: 加密 |
| send | send_pk | send_sk | 我→任何人: 签名 |

两个密钥对完全无关。一个泄露不影响另一个。这实现了「机密性」与「认证性」的分离。

### 3.3 算法选型

| 用途 | 算法 | 标准化程度 |
|:---|:---|:---|
| 密钥封装 (KEM) | ECDH (Curve25519) | RFC 7748 |
| 对称加密 | AES-256-GCM | NIST SP 800-38D |
| 签名 | Ed25519 | RFC 8032 |
| 哈希 | BLAKE3 | 广泛使用 |
| 后量子 (身份密钥) | Kyber-1024 | NIST PQC 标准 |
| 零知识证明 | zk-SNARKs (Groth16) | 成熟实现 |
| 双棘轮 | Signal Protocol 变体 | 广泛部署 |

> 所有密码学原语均基于现有标准实现，无需新发明。

### 3.4 线下邀请

```
邀请流程:
  ① Alice 与 Bob 线下见面
  ② Alice 生成邀请凭证: {bob_pre_pk, inviter_id, seed_credit, timestamp}
  ③ Alice 用 send_sk 签名 → invite_sig
  ④ Alice 通过二维码/蓝牙/NFC 传输: invite + invite_sig + Alice当前地址
  ⑤ Bob 首次上线时提交此凭证 → 链上记录 Bob 被 Alice 担保
```

**约束:**
- 多重担保: 需 ≥ 3 个不同来源邀请者共同担保
- 邓巴数上限: 每人有效担保人数受限于真实社交容量 (~150)
- 信誉传导: 邀请者对被担保人恶意行为承担连带信誉惩罚

## 五、协议层 · 双链架构

### 5.1 双链定义

```
┌─────────────────────────────────────────────────────┐
│                    明链 (资产链/结算层)               │
│                                                     │
│  记录内容:                                          │
│    · 用户注册信息 (chain_id + recv_pk + send_pk)    │
│    · 还款记录 (谁、何时、向资金池还了多少)            │
│    · 资金池余额                                     │
│    · 节点工资发放记录                                │
│    · 邀请担保关系 (zk遮蔽: 知道有担保，不知谁担保谁)   │
│                                                     │
│  可见性: 公开可查「谁还了钱」，但不知对应哪笔贷款      │
│  速度: 较慢 (日终清算级)                             │
│  资金: 所有资金沉淀于此                              │
│                                                     │
├─────────────────────────────────────────────────────┤
│                    暗链 (债务链/交易层)               │
│                                                     │
│  记录内容:                                          │
│    · 贷款申请记录                                   │
│    · 债务状态 (未还/已还/违约)                       │
│    · 匿名对手方化名 (一次性 zk 化名)                 │
│    · 交易结算指令                                   │
│                                                     │
│  可见性: 外部只见「有借贷发生」，不见借贷者身份        │
│  速度: 极快 (高频生成借条)                           │
│  资金: 无资金本体，仅记录借条                        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 5.2 四步交易闭环

```
第1步【暗链借贷】
  A 发起借贷申请 → 暗链根据匿名信用分审批 → 生成借条记录

第2步【明链放款】
  A 提交 ZK 证明 (证明自己有权借贷) → 明链验证合法 → 资金池划款至托管

第3步【暗链结算】
  双方确认交付 → 暗链生成结算指令 → 明链划转托管资金至 B

第4步【明链还款】
  A 向资金池回流资金 → 附带 ZK 还款证明 → 暗链更新债务状态
```

### 5.3 关键创新：还款 = 向资金池放贷

```
传统: 用户还债 → 债主资金增加 → 外部可见「有人还了钱」

本系统: 用户还债 → 用户向资金池发放一笔贷款
        → 资金池资产端和负债端同步增加
        → 资金池净值不变
        → 外部观察者无法区分「还债」与「大户向资金池注资」
```

这是一个语义翻转：还债在账本上表现为注资。外部攻击者看到的永远是「有人在向资金池投钱」，无法判断这是还债还是投资。

### 5.4 明链数据结构

```
明链核心: 每个用户拥有一个 UserBlock (链上数据块)
用户只需保留双密钥，链上查询即可重建一切。

UserBlock: chain_id → {
  recv_pk, send_pk,          ← 双密钥
  credit_score, trust_weight, ← 信誉
  friends[],                  ← 好友列表 (链上存储)
  guaranteed_of[],            ← 我担保了谁
  guaranteed_by[],            ← 谁担保了我
}

BondRegistry: bond_id → { inviter, invitee, amount, notes[], status }
CreditTree: Merkle(未花费信用票据)
PoolBalance: uint64
```

明链区块:
  BlockHeader:
    prev_hash, merkle_root, timestamp, epoch, consensus_data
  
  Transactions:
    RegisterUser    { chain_id, recv_pk, send_pk, zk_identity_proof }
    UpdateFriends   { sender, add[], remove[] }  ← 用户管理自己的好友列表
    GuaranteeBond   { inviter, invitee, bond_notes[], coefficient }  ← 公开担保
    RepayLoan       { zk_proof(我知道暗链某借条已结清), amount, timestamp }
    PayNodeSalary   { node_chain_id, zk_work_proof, amount, epoch }
```

### 5.5 暗链数据结构

```
暗链区块 (高频):
  BlockHeader:
    prev_hash, merkle_root, timestamp, epoch

  Transactions:
    RequestLoan     { zk_anon_id, zk_credit_proof, amount, term }
    IssueLoan       { loan_id, zk_anon_id_lender, zk_anon_id_borrower, amount }
    SettleTrade     { loan_id, zk_delivery_proof, settlement_cmd }
    UpdateDebt      { loan_id, zk_repay_proof, new_status }

  关键: 所有身份字段均为 zk 化名，与明链身份无直接关联
  
  每笔暗链交易附带:
    inkwell_params { time_seed, amount_fragments, relay_path }
    用于后续混沌结算
```

## 六、协议层 · 混沌结算池 (Inkwell)

### 6.1 问题

攻击者拥有明链全部数据 (还款记录) + 暗链全部数据 (借贷记录)，试图关联两者 → 找出「谁借了哪笔钱」。

### 6.2 三把锁

```
┌─────────────────────────────────────────────────┐
│  锁1: 时间随机化                                 │
│                                                 │
│  还款不在借贷到期日立即执行                       │
│  随机分散在 1~90 天内，分 3~7 次不等额执行       │
│  每次还款的时间由借贷时双方协商的 seed 决定       │
│                                                 │
├─────────────────────────────────────────────────┤
│  锁2: 金额拆封                                   │
│                                                 │
│  整额债务 (如 1000 元) 拆成随机碎片:              │
│    { 37.21, 12.09, 451.83, 298.07, 200.80 }    │
│  每片金额、片数、顺序均为随机                     │
│  明链上的还款记录是一堆无规律的碎数               │
│                                                 │
├─────────────────────────────────────────────────┤
│  锁3: 来源伪装 (接力回流)                        │
│                                                 │
│  还款碎片不直接从借款人账户流回资金池              │
│  可经过其他用户账户 "代付" 接力                   │
│  外部看到: 用户X还了37.21, 用户Y还了12.09...    │
│  无法确定这些碎片属于同一笔贷款                   │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 6.3 安全性

```
攻击者的任务:
  在 N 个用户的 M 笔还款碎片中
  找出哪些碎片组成了一笔特定的暗链贷款

复杂度:
  组合数 = C(M, k) × N!
  即使 N=1000, 实际不可行

效果:
  攻击者拥有双链全量数据
  仍无法将暗链借贷 ID 与明链用户身份关联
  概率等于在 N 个用户中随机猜测 (1/N)
```

### 6.4 实现参数

type InkwellParams struct {
    // 锁1: 时间随机化

## 七、协议层 · 经济体系

### 7.1 核心规则

**全局交易量恒等为零。** 货币由交易行为创造和销毁。

### 7.2 信用额度

| 规则 | 公式/说明 |
|:---|:---|
| 计算方式 | 信用限额 = 历史成功还款总额 × 系数 (≤1.0) |
| 种子期 | 新用户从资金池借贷 (上限小额)，邀请者担保 |
| 成熟期 | 还清种子贷款后自动解锁，邀请者无责 |
| 跑路约束 | 系数 ≤ 1.0 → 跑路净收益 ≤ 0，经济上不划算 |

### 7.3 资金池

```
资金来源: 每笔交易抽取固定比例

┌─────────────────────────────────────┐
│              资金池                  │
│                                     │
│  收入:                              │
│    地面层交易手续费 (日常商业)       │
│    游戏充值分账 (基础设施税)         │
│                                     │
│  支出:                              │
│    正式节点工资 (路由+记录, 月结)    │
│    非正式节点小费 (按单结算)         │
│    种子借贷池 (新用户首次借贷)       │
│    坏账准备金 (递归追偿到顶兜底)     │
│    社区公共物品 (基础设施维护)       │
└─────────────────────────────────────┘
```

### 7.4 经济闭环

```
交易抽成 → 资金池 → 矿工工资 → 矿工与用户交易 → 交易抽成 (重新注入)
    ↑                                                      │
    └──────────────────────────────────────────────────────┘

关键:
  矿工不是成本项，是内部消费者
  GDP 驱动节点数，不依赖人口增长
  全网正式节点数 ≤ 资金池月均收入 ÷ 单个节点最低工资
```

### 7.5 递归穿透式追偿

```
用户 N 违约 (欠款 = X):

① 系统从 N 自身余额扣款
② N 余额不足 → 没收担保人为 N 质押的担保金 (按比例)
③ 担保金不足 → 从担保人自身余额扣款
④ 担保人余额不足 → 递归向上追溯担保人的担保人
⑤ 递归至:
   · 某一级全额赔付 → 终止
   · 追偿链抵达种子节点且余额不足 → 资金池坏账准备金吸收

连带惩罚:
  质押金没收 + 信任权重扣减 (直接担保人最重)
```

## 八、协议层 · ZK 信任锚点

### 8.1 用途

允许用户在暗域匿名行事时，证明自己「在明域拥有良好信誉」，但不暴露明域身份。

### 8.2 证明类型

| 证明 | 输入 | 输出 | 用途 |
|:---|:---|:---|:---|
| **身份证明** | send_pk, 担保人签名 | zk_proof | 证明「被≥3人担保」 |
| **信用证明** | 还款历史 | zk_credit_proof | 证明「信誉分>阈值」 |
| **还款证明** | 暗链借条ID | zk_repay_proof | 证明「某借条已结清」 |
| **工作量证明** | 转发/存储记录 | zk_work_proof | 节点领工资凭证 |

### 8.3 证明生成/验证

```
生成 (用户本地):
  ① 收集原始数据 (担保人签名、还款记录...)
  ② 构造 zk 电路 (Groth16)
  ③ 生成证明 + 公开输入
  ④ 提交到链上

验证 (链上或任意节点):
  ① 接收 proof + public_inputs
  ② 运行验证器 (常数时间, ~3ms)
  ③ 通过 → 记录到链上; 失败 → 拒绝
```

## 十一、分布式随机信标 (应用层基础设施)

### 11.1 用途

在无信任环境下生成不可预测的随机数。用于游戏掉落、素材铸造、抽奖等。是去中心化应用的基础设施。

### 11.2 多贡献者随机协议

```
前提: K 个参与者需要共同生成一个随机数

① 提交阶段:
   每个参与者 i 生成 random_seed_i (32B)
   计算 commitment_i = Hash(random_seed_i || nonce_i)
   广播 commitment_i 到链上

② 揭示阶段 (提交阶段结束后):
   每个参与者广播 random_seed_i + nonce_i
   验证 Hash(random_seed_i || nonce_i) == commitment_i

③ 聚合:
   最终随机数 = Hash(random_seed_1 || random_seed_2 || ... || random_seed_K)
   
④ 安全保证:
   只要至少一个参与者是诚实的
   最终随机数就不可预测
   任何参与者无法单方面控制结果
```

## 十三、已知局限

| 局限 | 说明 |
|:---|:---|

| 还款时序残留 | Inkwell处理后关联概率极低但非绝对消除 |
